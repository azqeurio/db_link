# PR 5: Bayer-domain model preprocessor and OM-5 Mark II ISO series

## Scope and decision

PR 5 implements the isolated model-input path proposed by PR 4. It does not connect RAW-AI to the production editor or change the existing LibRaw display pipeline.

The project owner explicitly approved treating the supplied `OM-5MarkII` series as `OM-5`. The policy is deterministic across decoder versions: both rawpy/LibRaw 0.21.4 and Android LibRaw 0.22.0 use the `OM-5` color matrix from LibRaw 0.22.0 `colordata.cpp`. The source and effective camera names, matrix source, and fallback status are recorded. Unknown rank-deficient cameras remain rejected.

## Dataset

- 12 external ORF files, ISO 200, 400, 400, 800, 1000, 2000, 4000, 6400, 8000, 10000, 12800, and 25600.
- Every sample is 5240x3912 visible RGGB, identity orientation, white level 4095.
- Per-file black levels are preserved; they range from uniform `[254,254,254,254]` to channel-specific `[226,230,224,228]`.
- Center references were generated at 256, 512, and 1024 pixels: 36 crops total.
- RAW files and tensor binaries remain external. `pr5-om5-markii-dataset.json` records source and tensor SHA-256 hashes.
- The series has no clean ground-truth pair and cannot support absolute denoising-quality claims.

## Implemented contract

1. Extract the visible Bayer mosaic before `dcraw_process`.
2. Normalize CFA sites with `(sample - black[channel]) / (white - black[channel])`.
3. Apply camera RGB to linear Rec.2020 on the four RGGB planes.
4. Reassemble the transformed Bayer mosaic.
5. Demosaic with Malvar2004 and reflect boundary handling.
6. Clip to `[0,1]`.
7. Round through IEEE-754 binary16 and widen to float32 for the current FP32 oracle.
8. Use condition `min(ISO,65535)/6400`; `cond=0` remains an explicit numerical-oracle mode.

The Android implementation is `RawModelDomainPreprocessor`; camera aliasing is isolated in `RawCameraColorMatrixPolicy`. Debug-only JNI exposes the unpacked visible Bayer crop and metadata without changing production buffers.

## Verification

- Deterministic 8x8 JVM fixture generated by the independent Python pipeline: exact binary16-value match.
- All 12 physical-device 256x256 tests on Samsung SM-S936W / Android 16 / arm64-v8a: PASS.
- Across the 12 samples, RMSE ranged from `3.53e-7` to `1.22e-6`; maximum absolute error was at most `4.88e-4`.
- ISO 25600 1024x1024 physical-device test: PASS in 3.259 seconds, RMSE `2.84e-7`, maximum absolute error `1.22e-4`.
- Black-level extraction was corrected to use LibRaw's scalar `black` plus per-channel `cblack`; using `cblack` alone incorrectly returned zero for this camera.

## Remaining boundary

This proves Android preprocessing parity for one RGGB, identity-orientation camera family and the approved OM-5 alias. Non-RGGB CFA layouts, rotated active areas, unsupported-camera behavior on device, bounded full-resolution source/sink memory, accelerated FP16 execution, and production editor/export integration remain separate work.

The SuperLight/Light/Standard comparison tool and dataset contract are ready, but the three upstream TorchScript weights and a clean target are not locally available. Model ranking is therefore still pending.
